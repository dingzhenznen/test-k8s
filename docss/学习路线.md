# 学 K8s 要不要看源码？结论先说：
**入门、中级使用、运维、开发业务容器，完全没必要啃源码；只有深耕底层、定制二次开发、排查疑难内核问题才需要读源码。**

## 一、不同阶段该不该看源码
### 1. 入门阶段（只会装集群、部署应用）❌ 不建议看源码
目标：会 `kubectl`、Deployment/Service/ConfigMap、PV、命名空间、集群搭建、排简单故障
- 源码极其庞大（Go 几十万行），一上来读极易劝退
- 优先掌握：
  1. K8s 整体架构：apiserver、etcd、kube-scheduler、kube-controller-manager、kubelet、容器运行时
  2. 对象生命周期、声明式 API、控制器调和（Reconcile）思想
  3. 网络、存储、调度基础原理

### 2. 中级（运维专家 / 容器开发 / CNI/CSI 插件开发）⚠️ 选择性浅读关键片段
适合人群：要做自定义控制器 Operator、排查诡异调度异常、网络卡顿、资源泄露、kubelet 异常等
不用通读全量源码，只针对性看核心组件一小块：
- kube-apiserver：认证鉴权、资源增删改查、etcd 存储逻辑
- kube-scheduler：预选、优选打分调度逻辑
- kubelet：Pod 生命周期、镜像拉取、容器启停、探针实现
- controller：Deployment/StatefulSet 控制器调和逻辑

### 3. 高级（K8s 二次开发、内核贡献、定制发行版、厂商底层研发）✅ 必须深入源码
场景：
- 给 K8s 提交 PR、修改原生策略
- 自研调度器、定制 kubelet 行为
- 深度定制云原生平台、排查集群内核级死锁/逻辑 Bug
此时完整梳理源码结构、数据流转是必修课。

## 二、不看源码，怎么吃透 K8s 原理？性价比更高
1. **官方文档 + 官方设计文档（KEP）**
   KEP 是功能设计初稿，比源码更容易理解设计初衷
2. 吃透「声明式 API + 控制器调和模式」（K8s 最核心思想）
3. 跟踪对象流转：`kubectl apply → apiserver → etcd → controller 感知变更 → 调谐创建/删除 Pod`
4. 看组件流程图、博客拆解文章、公开课剖析核心逻辑
5. 自己写简单 Operator（用 client-go），不用读源码也能懂控制器原理

## 三、如果决定读源码，怎么起步最省力（避坑路线）
1. 环境准备：熟悉 Go 基础，会简单 Go 调试
2. 仓库：`kubernetes/kubernetes`
3. 阅读顺序（由浅入深）
   1. client-go：客户端如何发请求操作资源
   2. apiserver 请求处理链路
   3. Deployment 控制器完整调和逻辑
   4. kubelet Pod 生命周期管理
   5. 调度器打分逻辑
4. 工具：断点调试单个组件，打断点跟踪一次创建 Pod 的完整调用链，比逐行啃高效得多

## 总结一句话
- **只想用好 K8s、做运维/业务开发：不用看源码**
- **想做云原生底层、Operator 深度开发、疑难问题深度排障：选择性精读核心源码片段即可，不用通读全部**
