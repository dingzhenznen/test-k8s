• 是。对你这个例子来说，如果只是镜像版本变了，通常只需要改 test-k8s/yaml/service/
  app.yaml:1 里的 image，然后重新 apply：

  kubectl apply -f test-k8s/yaml/service/app.yaml

  原因是：

  - Deployment 管 Pod 的模板
  - image 属于 Pod 模板的一部分
  - 你改了 image 后，Deployment 会触发滚动更新，逐步把旧 Pod 换成新 Pod
  - test-k8s/yaml/service/service.yaml:1 一般不用改，因为 Service 只是按 label 找
    Pod，并不关心镜像版本

  前提是这几个东西别变：

  - Deployment 生成出来的 Pod 仍然带 app: test-k8s
  - 容器监听端口仍然还是 8080
  - Service 的 selector 仍然能匹配到这些 Pod

  只有在这些情况才需要改 service.yaml：

  - Pod 的标签变了
  - 容器端口变了，targetPort 也要跟着改
  - 你想把 ClusterIP 改成 NodePort / LoadBalancer

  你这个当前配置下，单纯升级镜像，改 app.yaml 就够了。

  也可以不用改文件，直接改镜像：

  kubectl set image deployment/test-k8s test-k8s=ccr.ccs.tencentyun.com/k8s-
tutorial/test-k8s:v2

› ingress 也不需要改 是吗


• 一般也不需要改。

  如果只是 test-k8s 的镜像版本变了，比如 v1 -> v2，通常只改 Deployment 里的 image 即
  可，Service 和 Ingress 都不用动。

  原因是：

  - Deployment 负责把旧 Pod 换成新 Pod
  - Service 还是按标签选中这批 Pod
  - Ingress 还是把请求转发到同一个 Service

  链路还是这条：

  Ingress -> Service -> Pod

  只要下面这些不变，Ingress 就不用改：

  - Service 名字不变，比如还是 test-k8s
  - Service 端口不变
  - Ingress 转发规则不变，比如域名、路径不变
  - 应用对外提供的协议和端口不变

  只有这些情况才可能要改 Ingress：

  - 你把后端 Service 名称改了
  - Service 端口改了
  - 域名或路径规则改了
  - 原来是 HTTP，后来改成 HTTPS 或增加 TLS 配置

  所以你这个场景里，单纯换镜像版本，通常是：

  - 改 app.yaml
  - kubectl apply -f ...
  - service.yaml 不用改
  - ingress.yaml 也不用改

  一句话记：

  改镜像，通常只动 Deployment；访问链路对象 Service 和 Ingress 一般不动
